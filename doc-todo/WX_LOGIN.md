# DeepSeek 网站微信登录流程整理 (暂未实现)

## 文档目的

本文用于整理你观察到的 DeepSeek 网站微信登录流程，并说明它大概率对应的是哪一套微信官方能力、页面上二维码如何生成、扫码后为什么会跳到微信确认页，以及服务端最终如何完成登录。

## 结论先说

你抓到的链接：

```text
https://open.weixin.qq.com/connect/confirm?uuid=031xaAvF1oEoll2x
```

从形态上看，这不是公众号扫码事件回调，也不是你业务系统自己的回调地址，而是微信开放平台网站应用登录流程中的**微信侧确认页面**。

DeepSeek 这类网站大概率走的是：

1. 网站应用微信登录
2. OAuth 2.0 `authorization_code` 模式
3. PC 页面展示微信登录二维码
4. 用户手机微信扫码后，在微信客户端里看到确认授权页面
5. 用户确认后，微信把浏览器重定向回业务方配置的 `redirect_uri`
6. 业务后端用 `code` 换 `access_token`
7. 再根据 `access_token + openid` 获取用户信息
8. 业务系统签发自己的登录态

## 整体流程

### 1. 用户点击网站上的“微信登录”

业务前端不会直接生成你看到的 `connect/confirm?uuid=...` 链接。

标准做法是前端先发起微信开放平台的二维码登录请求，入口通常是：

```text
https://open.weixin.qq.com/connect/qrconnect?appid=APPID&redirect_uri=REDIRECT_URI&response_type=code&scope=snsapi_login&state=STATE#wechat_redirect
```

也可以通过 `wxLogin.js` 内嵌二维码组件，在自己页面中直接渲染二维码。

此时网站会准备好这些参数：

- `appid`：网站应用的微信开放平台 AppID
- `redirect_uri`：登录成功后回跳的业务回调地址
- `response_type=code`
- `scope=snsapi_login`
- `state`：防 CSRF 的随机串

## 2. 微信侧生成二维码登录会话

用户在 PC 端打开上述登录入口后，微信会为这次二维码登录创建一条独立的登录会话。

你抓到的：

```text
https://open.weixin.qq.com/connect/confirm?uuid=031xaAvF1oEoll2x
```

可以理解为该二维码登录会话在微信侧的确认页入口。这里的 `uuid` 大概率就是这次扫码登录会话的微信侧标识。

要注意：

- `connect/confirm` 不是第三方业务方自己实现的页面
- 它也不是微信开放给业务方主动调用的标准接口
- 它更像是扫码后由微信 App 内部拉起的确认授权页

也就是说，业务方通常接入的是 `qrconnect`，而不是手动去拼 `connect/confirm`

## 3. 用户用手机微信扫码

当用户拿手机微信扫 PC 页面上的二维码后，微信客户端会识别出这是一个“网站应用微信登录”的授权请求。

如果需要扫码确认，微信会在手机端拉起你看到的 `connect/confirm?uuid=...` 页面，让用户确认是否同意登录该网站。

在较新的桌面微信环境下，如果用户设备已登录符合条件的微信客户端，还可能命中“微信快速登录”，这时不一定需要扫码，也可能直接在本机确认登录。

## 4. 用户确认授权

用户在微信侧确认后，微信会把这次授权结果返回给登录流程。

如果用户同意授权，微信会将 PC 浏览器重定向到业务方预先配置的回调地址，也就是：

```text
redirect_uri?code=CODE&state=STATE
```

如果用户拒绝授权，则不会发生成功回调。

这里真正对业务系统有价值的是：

- `code`
- `state`

其中：

- `code` 是一次性授权票据
- `state` 用于校验这次回调是不是自己发起的登录请求

## 5. 后端用 code 换 access_token

业务后端收到回调后，会调用微信开放平台接口，用 `code` 换取授权信息。

这一阶段通常能拿到：

- `access_token`
- `expires_in`
- `refresh_token`
- `openid`
- 在满足条件时的 `unionid`

这一步是整个微信登录流程里最关键的服务端动作之一，因为只有后端持有 `AppSecret`，所以必须由后端完成。

## 6. 后端再获取用户基本信息

如果业务系统需要用户资料，例如：

- 微信昵称
- 头像
- 性别
- 地区

则后端还会继续调用用户信息接口，通过：

- `access_token`
- `openid`

获取用户基本信息。

在网站应用微信登录场景里，这一步通常能拿到：

- `nickname`
- `headimgurl`
- `sex`
- `province`
- `city`
- `country`
- `unionid`（适用时）

## 7. 业务系统建立自己的账号体系

拿到微信侧身份信息后，业务系统一般不会直接把微信的 `access_token` 当作自己站内的登录态使用，而是会：

1. 先查库，看该 `openid` 或 `unionid` 是否已绑定现有账号
2. 如果没有，则创建业务用户
3. 保存昵称、头像、`openid`、`unionid`
4. 签发自己系统的 Session / JWT / Cookie
5. 最终让用户以“DeepSeek 账号”的身份登录网站

所以从业务实现看，微信只负责：

- 身份确认
- 授权
- 返回用户标识和基础资料

真正的网站登录态，仍然是 DeepSeek 自己签发的。

## DeepSeek 侧可能的前后端职责划分

### 前端职责

- 点击“微信登录”按钮
- 生成或渲染微信登录二维码
- 保存本次登录的 `state`
- 等待微信登录回跳
- 回调成功后把站内登录态写入浏览器

### 后端职责

- 生成微信登录链接所需参数
- 校验 `state`
- 用 `code` 换 `access_token`
- 获取 `openid / unionid / 用户信息`
- 创建或更新本地用户
- 返回自己的登录凭证

## 为什么你会看到 connect/confirm

这个现象的本质是：

- 业务方发起的是二维码登录请求
- 微信侧内部把扫码会话映射到一个确认页
- 手机微信扫到二维码后，用户进入微信自己的确认授权页面

因此你抓到 `connect/confirm?uuid=...` 是正常的，但它不代表 DeepSeek 自己实现了一个“扫码授权页”。

更准确地说：

- DeepSeek 接的是微信开放平台网站登录
- `confirm` 页面是微信负责展示的
- DeepSeek 只处理发起登录和接收回调

## 和公众号扫码登录的区别

这套流程和“公众号带参数二维码 + 事件推送登录”不是一回事。

### DeepSeek 这种更像是

- 微信开放平台
- 网站应用登录
- OAuth 授权回调

### 公众号扫码登录更像是

- 公众号生成带参数二维码
- 用户扫码后触发 `SCAN` / `subscribe` 事件
- 服务器收到事件推送后知道“谁扫了哪个码”
- 再由业务系统自己决定如何绑定登录

两者最大区别在于：

- 网站应用登录是标准 OAuth 登录流程
- 公众号二维码登录更像业务方自己拼出来的扫码登录流程

## 如果你要复刻类似流程，需要什么条件

如果你想做出和 DeepSeek 类似的 PC 网站微信扫码登录，通常需要：

1. 微信开放平台开发者账号
2. 已审核通过的网站应用
3. `AppID` 和 `AppSecret`
4. 已配置并审核通过的授权回调域名
5. 后端回调接口
6. 本地用户体系与登录态签发能力

仅有公众号本身，通常不能直接复刻这套标准网站扫码登录流程。

## 对当前观察的合理推断

基于你提供的链接，可以做出这些相对稳妥的推断：

- DeepSeek 登录页先发起了微信开放平台网站登录请求
- 微信为此次扫码登录分配了一个登录会话标识
- 用户扫码后，微信 App 打开了 `connect/confirm?uuid=...` 进行授权确认
- 用户确认后，浏览器会回跳到 DeepSeek 自己配置的 `redirect_uri`
- DeepSeek 后端再用 `code` 完成换票、取用户资料、建立本地登录态

其中“`uuid` 是微信侧扫码会话标识”“`connect/confirm` 是微信内部确认页”这两点，是基于官方标准流程和实际链接形态做出的工程推断，不是微信开放文档里明确要求第三方直接调用的公开接口。

## 可直接复用的认知框架

你可以把 DeepSeek 的微信登录简单理解成下面这条链路：

```text
网站点击微信登录
-> 微信开放平台生成二维码登录会话
-> 用户扫码
-> 微信客户端弹出确认授权页
-> 微信重定向回业务回调地址
-> 后端用 code 换 token
-> 后端拉取微信用户资料
-> 网站签发自己的登录态
```

## 参考依据

以下结论主要依据微信开放平台官方文档整理：

- 网站应用微信登录走 OAuth 2.0 `authorization_code` 模式
- PC 端标准入口为 `connect/qrconnect`，作用域为 `snsapi_login`
- 用户授权后回调到 `redirect_uri?code=...&state=...`
- 后端再用 `code` 换 `access_token`
- 可进一步获取用户基本信息
- 新版桌面微信支持快速登录，可无需扫码直接确认

如果后续你要把这个流程真正落到当前项目里，建议直接分成两种方案评估：

1. 公众号二维码登录
2. 微信开放平台网站应用登录

对于“像 DeepSeek 一样的 PC 网站扫码登录”，优先考虑第 2 种。
