# 停止生成与继续生成方案设计

## 一、背景说明

当前项目已经具备流式输出能力，前端通过 `fetch + AbortController` 接收 SSE 分片，后端在 `/api/chat/send` 中流式返回模型内容。

但现状只具备以下能力：

- 可以从技术层中断当前请求
- 可以基于上一条用户消息重新生成回答
- 不能真正从“上次中断的位置”继续生成

这意味着：

1. 用户看到回答生成到一半时，理论上可以中断。
2. 但中断后的半截回答目前主要存在前端内存中，不属于正式会话记录。
3. 用户再次点击“重新生成”时，本质上是重新发起一次新回答，而不是接着上一次未完成内容继续写。

因此，如果要支持正式的“停止生成 / 继续生成”，必须把“中断态”和“未完成回答”纳入会话模型。

---

## 二、目标

本方案的目标如下：

- 提供明确的“停止生成”按钮
- 停止后保留已经生成的内容
- 提供“继续生成”按钮
- 继续生成时尽量从上次中断位置继续，而不是整段重答
- 刷新页面后仍能识别未完成回答
- 保持现有会话、消息列表、流式渲染逻辑可兼容演进

---

## 三、结论先行

### 3.1 是否能实现继续生成

可以实现。

但前提不是只新增一个 `continue=true` 参数，而是必须同时补齐以下能力：

1. 中断时保留未完成 assistant 消息
2. 后端能够识别该消息是否为“未完成”
3. 继续生成时能基于该未完成消息重建上下文
4. 流式过程中持续更新消息，而不是只在流结束后一次性入库

### 3.2 推荐方案

推荐采用“消息持久化占位 + 继续生成续写”的正规方案，而不是只靠前端把半截文本回传给后端。

原因如下：

- 可以支持刷新页面后继续生成
- 可以支持多端一致的会话状态
- 可以避免未完成回答只保存在前端内存
- 更符合正式聊天产品的数据模型

---

## 四、现状问题

### 4.1 当前停止生成的问题

当前前端虽然已经使用 `AbortController`，但“停止生成”主要是请求层取消，存在以下问题：

- 没有明确的产品级停止按钮
- 中断后的回答不一定持久化到数据库
- 页面刷新后，中断内容可能丢失
- 后端无法区分“正常结束”还是“用户主动停止”

### 4.2 当前继续生成的问题

当前的“重新生成”逻辑，本质是：

1. 找到上一条用户消息
2. 删除当前 assistant 回答
3. 重新向模型发起请求

这并不等于“继续生成”，因为：

- 它没有利用上一次已经生成的 assistant 半成品
- 它无法保证模型从原位置续写
- 它可能重复生成前面已经出现过的内容

---

## 五、推荐的正规方案

## 5.1 核心思路

将 assistant 的一次回答拆分为两个阶段：

1. 生成开始时，先创建一条 assistant 占位消息
2. 流式过程中持续更新这条消息的内容、推理内容、搜索状态和生成状态

这样在用户点击“停止生成”时，数据库中已经存在一条“未完成 assistant 消息”，后续点击“继续生成”时即可基于这条消息续写。

### 5.2 消息状态设计

建议为 `chat_message` 增加状态字段，至少包括：

- `status`
  - `streaming`：生成中
  - `stopped`：用户主动停止
  - `completed`：生成完成
  - `failed`：生成失败

- `generation_id`
  - 用于标识一次完整生成链路
  - 停止生成与继续生成可以归属同一条生成链路

- `continue_from_message_id`
  - 记录本次继续生成所依赖的上一条未完成 assistant 消息
  - 便于调试和后续排查问题

如果不想一次改太多，最小可行字段如下：

- `status`
- `generation_id`

---

## 六、数据库改造建议

### 6.1 `chat_message` 表建议新增字段

建议新增如下字段：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `status` | varchar(20) | 消息状态：`streaming/stopped/completed/failed` |
| `generation_id` | varchar(64) | 同一轮生成链路标识 |
| `continue_from_message_id` | bigint | 继续生成时关联的上一条 assistant 消息 |
| `is_final` | tinyint | 是否为当前最终版本，便于后续做版本管理 |

### 6.2 初期最小落地

如果希望先快速实现，第一期只加：

- `status`
- `generation_id`

已经足够支撑“停止生成 / 继续生成”。

---

## 七、接口设计建议

### 7.1 发送消息接口保留

继续保留：

- `POST /api/chat/send`

但请求体建议扩展：

```json
{
  "content": "用户输入内容",
  "is_deepthink": false,
  "is_search": false,
  "session_id": 1,
  "continue_from_message_id": null
}
```

说明：

- 普通新问题时，`continue_from_message_id` 为 `null`
- 继续生成时，`content` 可为空字符串或沿用原问题内容
- 后端重点依据 `continue_from_message_id` 判断是否进入续写逻辑

### 7.2 新增停止生成接口

建议新增：

- `POST /api/chat/messages/{message_id}/stop`

功能：

- 将目标 assistant 消息状态从 `streaming` 更新为 `stopped`
- 标记该消息已经被用户主动中断

说明：

- 严格来说，请求层 `abort` 只能中断浏览器连接
- 真正的业务状态变更仍建议通过接口落库

### 7.3 继续生成仍复用 `/api/chat/send`

不建议单独增加 `/continue`，因为本质上仍然是一次新的流式生成。

建议统一复用：

- `POST /api/chat/send`

通过 `continue_from_message_id` 区分普通生成和继续生成。

这样接口模型更统一，后续也更容易维护。

---

## 八、后端生成流程设计

### 8.1 普通生成流程

1. 校验会话归属
2. 先写入用户消息
3. 创建一条 assistant 占位消息，状态为 `streaming`
4. 调用模型开始流式生成
5. 每收到分片，持续更新 assistant 消息内容
6. 生成结束后，将状态改为 `completed`

### 8.2 停止生成流程

1. 前端点击“停止生成”
2. 前端调用 `AbortController.abort()`
3. 同时调用停止接口，或在流中断回调中补一个状态上报
4. 后端将对应 assistant 消息状态改为 `stopped`
5. 已经生成的内容继续保留

### 8.3 继续生成流程

1. 前端点击“继续生成”
2. 前端拿到当前 assistant 消息 `message_id`
3. 调用 `/api/chat/send` 并传 `continue_from_message_id`
4. 后端读取该 assistant 消息的现有内容
5. 将现有 assistant 内容作为上下文的一部分喂给模型
6. 通过提示词约束模型“从末尾继续，不要重复前文”
7. 流式生成新的增量内容
8. 将增量内容追加到原 assistant 消息，或写入新版本消息
9. 结束后将状态改为 `completed`

---

## 九、继续生成的两种实现方式

### 9.1 方案 A：原消息续写

做法：

- 继续生成时直接更新原 assistant 消息
- 新分片直接拼接到原 `content` 后面

优点：

- 用户看到的是同一条消息继续往下长
- UI 直观
- 数据结构简单

缺点：

- 审计时不容易区分“首次生成内容”和“继续生成追加内容”
- 如果续写失败，排查问题时粒度较粗

### 9.2 方案 B：新版本消息续写

做法：

- 原 assistant 消息保留，状态改为 `stopped`
- 继续生成时创建一条新的 assistant 消息，记录 `continue_from_message_id`
- 前端只展示最后一条有效版本

优点：

- 版本清晰
- 更方便回溯与调试
- 更适合后期做“版本切换 / 对比”

缺点：

- 前端显示逻辑更复杂
- 需要处理旧版本隐藏和新版本替换

### 9.3 本项目建议

第一期建议优先采用：

- **方案 A：原消息续写**

原因：

- 当前项目结构更适合快速落地
- UI 层不用引入消息版本管理
- 用户感知最自然

后续如果要做高级能力，再升级为版本化方案。

## 十、模型提示词建议

继续生成时，仅靠拼接历史消息还不够，建议增加系统提示，明确约束模型行为。

建议追加如下提示语义：

```text
你需要基于上一条 assistant 未完成回答继续输出。
要求：
1. 从已有内容的末尾自然续写。
2. 不要重复已经输出过的句子。
3. 不要重新从头回答。
4. 保持与已有语气、结构和语言一致。
```

如果启用了联网搜索或深度思考，也应继续沿用同一轮配置，避免续写前后风格突变。

---

## 十一、前端改造建议

### 11.1 输入区交互

输入区建议增加两种状态：

- 普通状态：显示“发送”
- 流式生成中：显示“停止生成”

行为建议：

1. 生成中禁止再次发送新问题
2. 点击“停止生成”后，当前回答保留
3. 停止后在当前 assistant 消息尾部显示“继续生成”按钮

### 11.2 消息卡片交互

assistant 消息建议根据状态显示不同操作：

- `streaming`：显示“停止生成”
- `stopped`：显示“继续生成”
- `completed`：显示“重新生成”
- `failed`：显示“重试”

### 11.3 页面刷新后的表现

刷新页面后，消息列表接口应返回消息状态。

如果最后一条 assistant 消息状态为：

- `streaming`
  - 说明上次生成异常中断
  - 前端应按 `stopped` 处理，展示“继续生成”

- `stopped`
  - 直接展示“继续生成”

- `completed`
  - 正常展示

---

## 十二、后端实现要点

### 12.1 不再只在完成后插入消息

这是本次改造的核心点。

当前逻辑是：

1. 流式生成
2. 全部结束
3. 一次性插入 `user` 和 `assistant`

改造后应变为：

1. 先插入 `user`
2. 先插入 assistant 占位消息
3. 流式过程中持续更新 assistant
4. 结束时更新状态

### 12.2 中断异常要单独处理

流式生成时要区分以下情况：

- 用户主动停止
- 浏览器断开连接
- 模型接口异常
- 服务端内部异常

建议状态映射如下：

- 用户主动停止：`stopped`
- 浏览器异常断开：`stopped`
- 模型接口失败：`failed`
- 正常完成：`completed`

### 12.3 继续生成时的上下文构造

继续生成时，建议读取：

- 最近历史 user / assistant 消息
- 当前未完成 assistant 消息的已有内容
- 原问题内容

然后按如下顺序组织上下文：

1. 历史对话
2. 当前轮用户问题
3. 当前 assistant 已生成内容
4. 系统续写约束提示

这样模型更容易从正确位置续写。

---

## 十三、边界问题

### 13.1 继续生成可能重复内容

这是大模型常见问题，即便补了 continue 语义，也不能完全保证 0 重复。

可接受的工程策略：

- 提示词约束“不要重复前文”
- 后端对续写首段做轻量去重
- 前端允许用户再次停止或重新生成

### 13.2 深度思考内容如何处理

如果当前项目保留 `reasoning` 流，继续生成时建议：

- 已有 `reasoning` 继续保留
- 新生成的 `reasoning` 继续追加
- 如果模型不再返回推理内容，也不应清空原值

### 13.3 搜索引用如何处理

如果当前轮开启联网搜索，继续生成时建议：

- 默认继续沿用本轮已有引用
- 新搜索结果可追加，但不应覆盖已有引用
- `search_status` 按最终状态更新

---

## 十四、分阶段实施建议

### 14.1 第一阶段

先完成可用版本：

1. assistant 占位消息落库
2. 流式过程中持续更新消息
3. 增加消息 `status`
4. 前端增加“停止生成”
5. 前端增加“继续生成”
6. 后端支持 `continue_from_message_id`

### 14.2 第二阶段

补强稳定性：

1. 停止接口单独落库
2. 刷新页面恢复未完成状态
3. 继续生成重复内容去重
4. 完善失败状态与重试逻辑

### 14.3 第三阶段

扩展高级能力：

1. assistant 多版本管理
2. 历史版本切换
3. 继续生成与重新生成并存
4. 操作日志和审计能力

---

## 十五、最终建议

对于当前项目，推荐的落地顺序是：

1. 先做“停止生成”
2. 同时改造 assistant 占位消息持久化
3. 再做“继续生成”
4. 最后再考虑“版本化重新生成”

原因是：

- 没有持久化的未完成 assistant 消息，就没有真正可靠的“继续生成”
- 只有请求层 `abort`，不算正式产品能力
- `continue` 语义必须依赖消息状态和消息内容落库，才能真正成立

---

## 十六、结论

“继续生成”可以实现，但不能只补一个 `continue` 参数。

真正可用的方案必须满足：

- 有正式的停止生成交互
- 中断内容可保留
- assistant 未完成消息可持久化
- 后端可以识别并续写未完成消息

对本项目来说，最合理的实现路径是：

- **流式占位消息持久化**
- **消息状态管理**
- **基于 `continue_from_message_id` 的续写机制**

这是一套可以正式上线的方案，而不是仅在当前页面内“看起来能继续”的临时实现。
