# 智能对话平台（复刻DeepSeek Chat）详细技术方案
## 一、方案概述
本技术方案围绕复刻DeepSeek Chat （网站地址：https://chat.deepseek.com/）核心功能展开，采用**前端React+SSE实时通信+IndexedDB缓存**、**后端Python+FastAPI+MySQL**的技术架构，对接DeepSeek系列大模型，实现手机号登录、流式对话、文件上传、对话历史管理等核心能力。技术选型贴合需求，严格遵循**SSE实时回复**、**Python+MySQL**、**DeepSeek模型对接**、**前端IndexedDB缓存**的核心要求，保证功能与体验与原站高度一致。

### （三）核心功能实现逻辑
#### 1. 登录/注册功能
- 手机号校验：输入框限制11位数字，实时校验格式，错误时置灰发送验证码按钮
- 验证码发送：调用后端接口触发60s倒计时，新手机号自动注册，老手机号直接登录
- 微信扫码登录：生成定时刷新二维码，轮询后端检查扫码状态，确认后完成登录
- 协议校验：未同意用户协议/隐私政策时，禁用所有登录按钮
- 登录态存储：登录成功后，将用户token、信息存入localStorage，后续接口请求携带token

#### 2. 实时对话（SSE流式回复）
- 建立SSE连接：通过`new EventSource(url)`连接后端SSE接口，url携带对话ID、用户ID、功能开关参数（深度思考/联网搜索）
- 流式数据接收：后端分块返回AI回复内容，前端实时追加至对话气泡，展示“打字机”效果
- 停止生成：点击停止按钮，关闭SSE连接，终止数据接收
- 异常处理：网络中断/接口异常时，展示错误提示，支持重新连接

#### 3. 对话历史管理
- 本地缓存（IndexedDB）：
  - 存储结构：`chatSessions`（会话列表：ID、标题、创建时间、更新时间）、`chatMessages`（消息列表：会话ID、角色、内容、时间）、`fileRecords`（文件记录：会话ID、文件信息）
  - 操作逻辑：新建对话时同步创建本地记录，发送消息后更新本地消息，删除/重命名会话时同步修改本地数据
- 后端同步：后台异步将本地对话历史同步至MySQL，保证数据持久化
- 列表展示：左侧对话列表按更新时间倒序展示，支持搜索筛选、hover操作（重命名/删除）

#### 4. 文件上传功能
- 上传逻辑：支持点击/拖拽上传，单文件≤100MB，格式限制PDF/图片/Excel
- 进度展示：上传时显示进度条，上传完成后在输入框展示文件列表，支持删除
- 数据关联：上传文件后，将文件信息存入IndexedDB，发送消息时携带文件ID，后端解析文件内容后传入大模型

#### 5. 功能开关逻辑
- 深度思考（R1）：开启后调用DeepSeek R1模型，回复展示推理链，支持展开/收起
- 联网搜索：开启后调用DeepSeek联网能力，回复标注来源链接，与文件上传互斥（开启时文件上传置灰）
- 主题切换：浅色/深色模式，主题配置存入localStorage，页面加载时同步应用

### （四）前端性能优化
1. 组件懒加载：通过`React.lazy()`拆分非核心组件（如设置页、帮助页），减少首屏加载资源
2. 渲染优化：使用`memo`缓存组件，避免无效重渲染；对话消息采用虚拟列表，优化大量消息时的滚动性能
3. 资源优化：CSS/JS文件压缩，图片资源采用懒加载，CDN加速静态资源加载
4. SSE优化：关闭非活跃连接，避免资源浪费；断线重连机制，保证流式回复稳定性

### （四）SSE流式回复实现逻辑
1. **协议规范**：后端接口响应头设置`Content-Type: text/event-stream; charset=utf-8`，保持长连接
2. **数据格式**：采用SSE标准格式，发送`data: {content:"片段内容"}\n\n`，前端通过`message`事件接收
3. **模型对接流程**：
   - 前端请求携带会话ID、用户输入、功能开关参数
   - 后端校验参数后，调用DeepSeek API，传入对话上下文、文件解析内容
   - DeepSeek API流式返回结果，后端实时转发至前端
   - 对话结束后，后端将完整消息存入MySQL，同步IndexedDB数据

### （五）大模型对接逻辑
1. **模型版本选择**：
   - 基础对话：DeepSeek-V3模型
   - 深度思考：DeepSeek-R1模型
   - 联网搜索：调用DeepSeek联网能力接口
2. **对接流程**：
   - 封装DeepSeek API调用工具，支持流式请求
   - 构建对话上下文：按会话ID获取历史消息，拼接为模型输入格式
   - 文件解析：调用文件解析接口，提取PDF/图片/Excel内容，拼接至输入参数
   - 联网搜索：开启开关时，模型自动调用联网能力，返回带来源的结果
3. **异常处理**：模型调用失败时，返回友好提示，记录错误日志

### （六）数据同步逻辑
1. **前端→后端**：
   - 新建/删除/重命名会话：实时同步至MySQL
   - 发送消息：消息发送后异步同步至MySQL
   - 文件上传：上传完成后同步文件记录至MySQL
2. **后端→前端**：
   - 登录后拉取全量会话历史，初始化IndexedDB
   - 后台定时同步（5分钟），保证前后端数据一致

## 五、安全与合规方案
1. **数据加密**：用户手机号、密码采用AES-256加密存储，接口传输采用HTTPS协议
2. **登录安全**：验证码60s倒计时、10分钟过期，同一手机号24小时最多5次验证码发送，防止刷取
3. **接口安全**：所有接口校验token，无权限请求直接拒绝；添加IP限流，防止接口滥用
4. **文件安全**：上传文件扫描恶意代码，设置文件访问权限，禁止执行脚本文件
5. **合规要求**：页面底部展示ICP备案号，用户协议/隐私政策符合国家网络安全法规

## 六、风险与应对措施
1. **SSE连接中断**：实现断线重连机制，前端检测断开后自动重新建立连接
2. **大模型调用失败**：配置重试机制，失败3次后返回错误提示，记录日志
3. **MySQL性能瓶颈**：分库分表（按用户ID分片），添加Redis缓存高频数据
4. **文件存储溢出**：定期清理无用文件，配置单用户文件存储上限
5. **前端IndexedDB存储满**：设置存储上限，超过时提示用户清理本地缓存
