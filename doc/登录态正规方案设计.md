# 登录态正规方案设计

## 一、背景说明

当前项目的登录态方案为：

- 前端将 `token` 和 `user` 保存到 `localStorage`
- 后端签发一个有效期为 7 天的 JWT
- 前端请求时通过 `Authorization: Bearer <token>` 携带登录态
- 退出登录时以前端清理本地缓存为主，服务端没有真正的会话失效机制

这个方案适合开发联调阶段，但不适合作为正式生产方案长期使用。

主要问题如下：

1. `localStorage` 中的令牌容易受到 XSS 攻击影响。
2. Access Token 有效期过长，泄露后风险窗口过大。
3. 服务端没有 Refresh Token 管理机制，无法做真正的续期和撤销。
4. 退出登录后旧 Token 在过期前仍可能继续使用。
5. 前端只判断本地是否存在 Token，缺少启动阶段的服务端校验。

---

## 二、核心问题：Access Token 变短会不会导致频繁登录

不会。  

正规的方案不是“把 Access Token 缩短，然后让用户反复手动登录”，而是：

- **Access Token** 负责接口鉴权，生命周期很短，例如 15 分钟到 2 小时
- **Refresh Token** 负责自动续签 Access Token，生命周期较长，例如 7 天到 30 天

用户的真实体验通常是：

1. 用户首次登录一次。
2. 前端后续访问接口时使用短效 Access Token。
3. Access Token 过期后，前端自动调用刷新接口。
4. 服务端校验 Refresh Token 合法后，重新签发新的 Access Token。
5. 用户无需重新输入账号密码。

只有在以下场景下，用户才需要重新登录：

- Refresh Token 过期
- 用户主动退出登录
- 服务端撤销了当前设备会话
- 检测到风险行为，强制要求重新认证

所以，**短效 Access Token 不等于频繁登录**。  
它的本质是：把“高频接口鉴权”和“长期会话保持”拆成两个层次。

---

## 三、推荐的正式方案

### 3.1 令牌设计

建议采用双令牌机制：

- **Access Token**
  - 用途：接口鉴权
  - 生命周期：15 分钟
  - 携带方式：前端内存中保存，请求头 `Authorization: Bearer <token>`

- **Refresh Token**
  - 用途：刷新 Access Token
  - 生命周期：7 天到 30 天
  - 携带方式：写入 `HttpOnly Cookie`

### 3.2 为什么这样设计

这样设计的目的如下：

1. Access Token 放内存，不持久化到 `localStorage`，可以降低 XSS 窃取风险。
2. Refresh Token 放 `HttpOnly Cookie`，前端 JavaScript 无法直接读取。
3. Access Token 即使泄露，风险窗口也很短。
4. Refresh Token 可以绑定服务端会话记录，实现可撤销、可下线、可风控。

---

## 四、推荐的会话模型

服务端不要只签发一个静态 JWT，而应当维护“设备级会话”。

建议新增一张会话表，例如：

### 4.1 `user_session` 表建议字段

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `user_id` | bigint | 用户 ID |
| `refresh_token_hash` | varchar | Refresh Token 的哈希值，不直接存明文 |
| `device_name` | varchar | 设备名称，可选 |
| `ip_address` | varchar | 最近登录 IP |
| `user_agent` | varchar | 浏览器标识 |
| `status` | tinyint | 1=有效，0=已失效 |
| `expire_time` | datetime | Refresh Token 过期时间 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |
| `last_active_at` | datetime | 最后活跃时间 |

### 4.2 存储原则

必须遵守以下原则：

1. 不保存明文 Refresh Token。
2. 数据库存储 Refresh Token 的哈希值。
3. 服务端校验时，对用户提交的 Refresh Token 做哈希后再比对。
4. 支持单设备注销和全部设备注销。

## 五、完整登录流程

### 5.1 登录流程

1. 用户输入手机号和密码登录。
2. 服务端校验账号密码。
3. 服务端生成：
   - 短效 Access Token
   - 长效 Refresh Token
4. 服务端将：
   - Access Token 返回给前端响应体
   - Refresh Token 写入 `HttpOnly Cookie`
5. 服务端记录 `user_session`
6. 前端将 Access Token 保存在内存状态中

### 5.2 刷新流程

1. 前端请求接口发现 Access Token 已过期。
2. 前端调用 `/api/auth/refresh`。
3. 浏览器自动携带 `HttpOnly Cookie` 中的 Refresh Token。
4. 服务端校验 Refresh Token 是否存在、是否过期、是否已撤销。
5. 校验通过后：
   - 签发新的 Access Token
   - 建议轮换新的 Refresh Token
   - 更新数据库中的会话记录
6. 前端拿到新的 Access Token 后重试原请求

### 5.3 退出登录流程

1. 前端调用 `/api/auth/logout`
2. 服务端根据 Refresh Token 定位当前会话
3. 服务端将该会话置为失效
4. 服务端清除 Cookie
5. 前端清理内存中的 Access Token 和用户信息

### 5.4 全部设备退出

可以扩展 `/api/auth/logout-all`：

1. 服务端根据当前用户 ID 查询所有有效会话
2. 全部置为失效
3. 清除当前设备 Cookie

## 六、前端存储建议

### 6.1 不建议的方式

不建议继续使用：

- `localStorage` 保存 Access Token
- `sessionStorage` 长期保存 Refresh Token
- 浏览器可读 Cookie 存 Refresh Token

### 6.2 推荐方式

推荐如下：

- **Access Token**
  - 保存位置：前端内存
  - 例如：React Context、Zustand Store
  - 页面刷新后通过 `/auth/refresh` 重新获取

- **Refresh Token**
  - 保存位置：`HttpOnly Cookie`
  - 属性建议：
    - `HttpOnly`
    - `Secure`
    - `SameSite=Lax` 或 `SameSite=Strict`
    - 生产环境必须限制域名和路径

### 6.3 页面刷新后的行为

页面刷新后，前端内存中的 Access Token 会丢失，这是正常设计。

前端启动时应执行：

1. 调用 `/api/auth/refresh` 或 `/api/auth/me`
2. 如果刷新成功，则恢复登录态
3. 如果刷新失败，则跳转登录页

这也是正规方案中的标准行为。

### 6.4 HttpOnly Cookie 是如何写入的

在正式方案中，`HttpOnly Cookie` 必须由服务端通过响应头写入，前端 JavaScript 不负责写入。

标准做法如下：

1. 用户登录成功后，服务端生成 Refresh Token。
2. 服务端在响应中通过 `Set-Cookie` 写入浏览器。
3. Cookie 配置为：
   - `HttpOnly`
   - `Secure`
   - `SameSite`
   - `Path`
   - `Max-Age` 或 `Expires`
4. 浏览器收到响应后自动保存该 Cookie。
5. 前端 JavaScript 无法通过 `document.cookie` 读取该值。

这样设计的主要目的是降低 XSS 窃取 Refresh Token 的风险。

### 6.5 HttpOnly Cookie 中写入的内容是什么

写入 `HttpOnly Cookie` 的内容通常是 **Refresh Token 明文**，而不是：

- 用户信息
- Access Token
- 明文密码
- 数据库主键直接拼接值

推荐做法如下：

1. 浏览器 Cookie 中保存随机生成的 Refresh Token 明文。
2. 服务端数据库中只保存该 Refresh Token 的哈希值。
3. 服务端校验时，对客户端提交的 Refresh Token 先做哈希，再与数据库比对。

这样即使数据库泄露，也不会直接暴露可用的 Refresh Token 明文。

示意如下：

- 浏览器 Cookie：`refresh_token=<随机长字符串>`
- 数据库存储：`refresh_token_hash=<哈希值>`

## 六点五、关闭浏览器后的登录态行为

关闭浏览器后，登录态通常**不会立即失效**，前提是 Refresh Token Cookie 仍然有效。

原因如下：

1. Access Token 保存在前端内存中。
2. 浏览器关闭后，内存中的 Access Token 会丢失。
3. 但 Refresh Token Cookie 如果设置了 `Max-Age` 或 `Expires`，则不会因为关闭浏览器立刻删除。
4. 用户再次打开浏览器并进入系统时，前端会自动调用刷新接口。
5. 浏览器会自动携带仍然有效的 Refresh Token Cookie。
6. 服务端校验通过后重新签发新的 Access Token。
7. 用户表面上仍然保持登录状态。

因此，正规方案下的真实体验通常是：

- 刷新页面不会直接掉登录
- 关闭浏览器再打开通常也不会直接掉登录
- 只有在以下情况才需要重新登录：
  - Refresh Token 过期
  - 用户主动退出登录
  - 浏览器清除了 Cookie
  - 服务端将当前会话失效

如果希望实现“关闭浏览器就必须重新登录”，可以改成不设置持久化过期时间，让 Refresh Token 使用 Session Cookie。  
但这会明显降低用户体验，一般不建议作为默认方案。

## 七、接口设计建议

建议新增或改造以下接口：

### 7.1 `POST /api/auth/login`

功能：

- 用户登录
- 返回 Access Token
- 写入 Refresh Token Cookie

响应示例：

```json
{
  "success": true,
  "code": 200,
  "message": "登录成功",
  "data": {
    "accessToken": "xxx",
    "expiresIn": 900,
    "user": {
      "id": 1,
      "nickname": "张三",
      "phone": "13800000000",
      "avatar": ""
    }
  }
}
```

### 7.2 `POST /api/auth/refresh`

功能：

- 使用 Cookie 中的 Refresh Token 换取新的 Access Token

响应示例：

```json
{
  "success": true,
  "code": 200,
  "message": "刷新成功",
  "data": {
    "accessToken": "xxx",
    "expiresIn": 900
  }
}
```

### 7.3 `GET /api/auth/me`

功能：

- 获取当前登录用户信息
- 用于前端应用初始化时恢复用户状态

### 7.4 `POST /api/auth/logout`

功能：

- 注销当前设备会话
- 清除 Refresh Token Cookie

### 7.5 `POST /api/auth/logout-all`

功能：

- 注销当前用户全部设备会话

## 八、安全建议

### 8.1 Cookie 配置建议

生产环境中 Refresh Token Cookie 建议配置：

- `HttpOnly=true`
- `Secure=true`
- `SameSite=Lax`
- `Path=/api/auth`

如果前后端跨域，需要结合实际部署评估：

- 是否需要 `SameSite=None`
- 是否必须启用 HTTPS
- 是否需要单独的认证域名

### 8.2 CSRF 防护

由于 Refresh Token 放在 Cookie 中，必须考虑 CSRF。

建议方案：

1. 登录成功后同时下发 CSRF Token
2. 前端在 `POST /PUT /DELETE` 请求头中携带 `X-CSRF-Token`
3. 服务端校验 Cookie 中会话与请求头 CSRF Token 是否匹配

如果刷新接口只接受同站请求，也可以结合：

- `SameSite=Lax/Strict`
- Origin / Referer 校验

### 8.3 Refresh Token 轮换

建议刷新时执行 Refresh Token 轮换：

1. 刷新成功后签发新的 Refresh Token
2. 旧 Refresh Token 立即失效
3. 如果检测到旧 Token 被重复使用，说明可能存在盗用风险
4. 可直接让该设备或该用户全部会话失效

### 8.4 风控能力

正式方案中建议保留以下能力：

- 记录设备信息
- 记录最近访问 IP
- 支持会话列表查看
- 支持单端踢下线
- 支持敏感操作二次验证

## 九、结合本项目的改造建议

当前项目技术栈：

- 后端：FastAPI
- 前端：React + TypeScript

建议按下面顺序改造。

### 9.1 后端改造步骤

1. 新增 `user_session` 表
2. 登录接口改为签发双令牌
3. 新增 `/api/auth/refresh`
4. 新增 `/api/auth/me`
5. 改造 `/api/auth/logout`
6. 增加 Refresh Token 哈希存储和轮换逻辑
7. 增加 Cookie 设置逻辑
8. 增加 CSRF 防护

### 9.2 前端改造步骤

1. 删除 `localStorage` 中对 Access Token 的长期依赖
2. 将 Access Token 改为保存在 Zustand Store 或内存状态中
3. 应用启动时先调用 `/api/auth/refresh` 或 `/api/auth/me`
4. axios 增加统一 401 和自动刷新逻辑
5. 刷新失败后再跳回登录页
6. 用户信息改为通过服务端恢复，而不是只依赖本地缓存

### 9.3 过渡方案

如果希望降低一次性改动风险，可以分两步进行：

#### 第一步

- 保留现有 Bearer Token 结构
- 新增 `/auth/me`
- 缩短 Access Token 时长
- 增加 `/auth/refresh`
- Refresh Token 改为 Cookie

#### 第二步

- 去掉 `localStorage` 中的 Token 持久化
- 前端改为内存 Access Token
- 增加会话管理、单端退出、全部退出
- 增加 CSRF 和 Refresh Token 轮换

## 十、推荐的最终落地方案

本项目推荐采用以下正式方案：

1. Access Token：15 分钟，保存在前端内存中
2. Refresh Token：7 天，保存在 `HttpOnly Cookie`
3. 前端启动时自动调用 `/api/auth/refresh` 恢复登录态
4. 服务端维护 `user_session` 表
5. Refresh Token 使用哈希存储
6. 刷新时执行 Refresh Token 轮换
7. 登出时服务端失效当前会话
8. 后续扩展全部设备退出和会话管理

这是兼顾安全性、用户体验和工程复杂度的平衡方案。

---

## 十一、结论

关于“Access Token 变短会不会导致频繁登录”，答案是：

**不会，只要配套引入 Refresh Token 自动续签机制，用户通常感知不到频繁登录。**

真正正规的登录态方案应当做到：

- 短效 Access Token
- 长效 Refresh Token
- Refresh Token 放 `HttpOnly Cookie`
- 服务端维护可撤销会话
- 前端自动恢复登录态
- 退出登录可真正失效

对于当前项目，建议将现有的“长效 JWT + localStorage”逐步升级为上述方案。
